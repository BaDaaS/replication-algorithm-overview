# 0088: Bitcoin-NG (Next Generation)

## Historical context

Ittay Eyal, Adem Efe Gencer, Emin Gun Sirer, and Robbert van
Renesse published "Bitcoin-NG: A Scalable Blockchain Protocol"
at NSDI 2016. Bitcoin-NG separates Bitcoin's two coupled
functions (leader election and transaction serialisation) into
two independent block types:

- *Key blocks.* Carry only a leader-election proof-of-work; one
  per epoch. The miner who finds a key block becomes the leader.
- *Microblocks.* Generated by the leader at regular intervals
  (e.g., 10 seconds). Carry transactions but require no PoW;
  signed by the leader.

This is conceptually similar to Multi-Paxos (module 0023):
expensive coordination is amortised across many fast
operations once a leader is elected.

The motivation was throughput: Bitcoin's 10-minute blocks limit
practical throughput to ~7 transactions per second. By moving
transactions out of PoW blocks, Bitcoin-NG can serve thousands
of transactions per second while keeping the security
properties of the longest-chain rule intact.

## System and threat model

- **Network.** Bounded-delay (PSS-style).
- **Failures.** Computational majority of honest hashing power.
- **Cryptography.** Standard PoW for key blocks, ECDSA
  signatures for microblocks.
- **Goal.** Higher throughput than Nakamoto without sacrificing
  security.

## Theory

### Block types

| field            | key block  | microblock                |
| ---------------- | ---------- | ------------------------- |
| frequency        | ~10 min    | ~10 sec                   |
| PoW required     | yes        | no                        |
| signed           | no         | yes (by leader)           |
| carries txs      | no (or few) | yes                      |
| determines leader | yes       | no                        |

The current leader is determined by the most recent key block.
That leader signs microblocks until the next key block is
mined.

### Leader election by PoW

Mining a key block proves computational work. The key-block
miner is elected leader; their public key is included in the
block header. They become the *micro-leader* until the next key
block is mined by someone else.

This is essentially Nakamoto consensus on key blocks; the
longest-chain rule applies to key blocks. The transaction
content is carried by microblocks signed by the leader.

### Equivocation: poison transactions

A leader could equivocate (sign two conflicting microblocks at
the same height) and split the chain. Bitcoin-NG handles this
with *poison transactions*: any subsequent miner who detects
equivocation can include a poison tx that destroys the
malicious leader's reward.

This is similar to slashing in PoS chains (Casper FFG, module
0123) but uses the leader's own signature as the slashing
proof.

### Comparison: Bitcoin vs Bitcoin-NG vs PoS leader rotation

| property         | Bitcoin (longest chain) | Bitcoin-NG     | Algorand-style PoS |
| ---------------- | ----------------------- | -------------- | ------------------ |
| leader election  | every block (PoW)       | key blocks (PoW) | per round (VRF)  |
| tx throughput    | ~7 tx/s                 | ~thousands tx/s | ~thousands tx/s   |
| equivocation     | not penalised           | poison tx      | slashing           |
| committee size   | full network            | leader only    | committee VRF     |
| safety in async  | probabilistic           | probabilistic  | probabilistic     |
| finality time    | ~1 hour                 | ~1 hour (key) + 10 s (micro) | seconds  |
| leader uptime    | n/a                     | required for live tx | required        |

The Bitcoin-NG design preserves Nakamoto's safety guarantees on
the key-block chain (so common prefix, chain growth, and chain
quality apply) while adding microblock-level throughput.

### Limitations

- *Leader can withhold.* A malicious leader can refuse to issue
  microblocks. Honest miners must wait for the next key block.
- *Bandwidth concentration.* The leader becomes a bandwidth
  bottleneck during their epoch.
- *Selfish mining still possible.* Withholding key blocks works
  the same as in Bitcoin (Eyal-Sirer 2014; module 0087).
- *No instant finality.* Microblocks are still subject to chain
  reorganisation if the leader's key block is itself orphaned.

### Subsequent influence

Bitcoin-NG influenced:

- *Hybrid Consensus* (Pass-Shi 2017, module 0091): committee
  rotation by PoW.
- *ByzCoin* (Kogias et al. 2016, module 0090): committee-based
  BFT after PoW selection.
- *Tendermint* (module 0055) and *HotStuff* (module 0056):
  adopt the Multi-Paxos style of leader-based pipeline, though
  with BFT rather than PoW for selection.

## Practice

- *Waves* (cryptocurrency, 2016 onward) initially used a
  Bitcoin-NG-inspired design.
- *Aeternity* (2018) adopted Bitcoin-NG with VRF-based block
  attribution.
- No major Bitcoin fork has adopted Bitcoin-NG, despite the
  throughput improvement; the change to dual block types is
  considered too invasive.

### Production-implementation notes

- Key block reward and microblock reward must be balanced. Eyal
  et al. propose 40% to the key-block miner and 60% to
  microblock fees, with a delayed split (60% paid to the next
  leader to discourage selfish mining of key blocks).
- Microblock signature verification must be fast; leader
  rotates every key block (~10 min).
- Poison-tx detection requires storing all signed microblocks
  for at least one epoch.

## Verifiability and circuit encoding

**tag: `partial`.**

Bitcoin-NG circuits encode two block types: PoW key-block
verification (SHA-256, like Bitcoin; see module 0083) and
ECDSA-signed microblock verification.

Microblock verification is dominated by signature checks (~1k
constraints per ECDSA verification with optimised circuits).
For ~600 microblocks per epoch, microblock cost is comparable
to one key-block PoW verification. ECDSA on secp256k1 is not
SNARK-friendly; a Bitcoin-NG SNARK light client would benefit
from substituting EdDSA on a SNARK-friendly curve.

## Known attacks and limitations

- *Leader silence.* A leader who stops producing microblocks
  stalls transaction throughput until the next key block.
- *DDoS on leader.* The current leader is publicly identified;
  attackers can target them.
- *Selfish key-block mining.* Eyal-Sirer attack (module 0087)
  applies unchanged.

## References

## Implementation notes

The crate provides `KeyBlock` and `Microblock` structs with the
fields required by the protocol: epoch number, parent hash,
leader public key (placeholder), and signed transactions.
Tests verify a leader can generate a sequence of microblocks
indexed by epoch, and that microblocks attach to the most
recent key block.
